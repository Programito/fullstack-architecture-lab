# Plan: Menu UX v2

Fecha: 2026-07-01  
Rama base: `main`  
Prioridad: A (correcciones rápidas) → B (diagnóstico imagen) → C (nueva funcionalidad)

---

## Contexto

Cinco mejoras identificadas en la página de gestión del menú tras la integración de imágenes Cloudinary y grupos de modificadores (commit `cc24b01`):

1. El panel de detalle de la derecha desaparece al hacer scroll.
2. La subida de imagen muestra error en frontend pero el backend no registra ningún fallo → el fallo está en Cloudinary.
3. No hay UI para crear nuevos grupos de modificadores (solo listar los existentes).
4. El health panel + chips de revisión siempre visibles ocupan espacio sin que el usuario las use activamente.
5. El filtro segmentado "Con imagen / Sin imagen" no aporta valor como filtro permanente.

---

## Bloque A — Correcciones de UX (sin dependencias)

### A1. Panel derecho: sticky al scroll

**Causa**: El `<aside>` en [menu-page.html:401](../../frontend/src/app/features/menu/pages/menu-page/menu-page.html#L401) tiene `self-start` pero no `sticky`. Como `<main>` tiene `min-h-dvh`, el scroll es de la página completa y el panel desaparece al bajar.

**Fix** — cambiar la clase del `aside`:

```diff
- class="theme-panel h-fit self-start rounded-xl border p-5 shadow-sm lg:block lg:h-[calc(100dvh-7rem)] lg:overflow-y-auto"
+ class="theme-panel self-start rounded-xl border p-5 shadow-sm lg:block lg:sticky lg:top-[7rem] lg:max-h-[calc(100dvh-7rem)] lg:overflow-y-auto"
```

- `sticky top-[7rem]`: se ancla bajo el header sticky (`7rem` ya se usa en el grid del mismo template).
- `max-h-` en lugar de `h-`: si el contenido es corto no fuerza altura completa.
- Se elimina `h-fit` (era redundante con `self-start` y conflictivo con la altura máxima).

**Archivos**: [menu-page.html](../../frontend/src/app/features/menu/pages/menu-page/menu-page.html)  
**Tests**: no requiere spec nuevo; los specs existentes del componente siguen verdes.

## Bloque B — Diagnóstico y fix de subida de imagen

### B1. Identificar el error real de Cloudinary

**Síntoma**: El frontend muestra "No se pudo subir la imagen. Inténtalo de nuevo." El backend no registra nada porque el fallo ocurre en el paso 3 (POST a `api.cloudinary.com`), que el backend no gestiona.

**Flujo de la subida**:
1. Frontend llama `POST /api/v1/restaurants/:id/products/image-upload-signature` → backend devuelve `{cloudName, apiKey, timestamp, folder, signature}`.
2. Frontend construye `FormData` con esos campos + el archivo.
3. Frontend hace `POST https://api.cloudinary.com/v1_1/{cloudName}/image/upload`.

Si el backend no arroja error, el paso 1 funciona. El fallo está en el paso 3.

**Pasos de diagnóstico**:

1. Con backend y frontend en marcha, abrir DevTools → pestaña Network → filtrar por `cloudinary.com`.
2. Intentar subir una imagen.
3. Observar la respuesta HTTP del endpoint de Cloudinary. Los errores más comunes:

| Respuesta Cloudinary | Causa |
|---|---|
| `{"error":{"message":"Invalid Signature"}}` | La clave en `.env` no coincide con la cuenta o el algoritmo de firma |
| `{"error":{"message":"Must supply api_key"}}` | `apiKey` no llega en el FormData |
| `{"error":{"message":"Cloud not found"}}` | `CLOUDINARY_CLOUD_NAME` en `.env` incorrecto |
| HTTP 401 / 403 | Credenciales expiradas o cuenta sin activar |

**Verificación de la firma** ([cloudinary-product-image-signer.ts](../../backend/src/restaurants/infrastructure/media/cloudinary-product-image-signer.ts)):

```typescript
const signatureBase = `folder=${folder}&timestamp=${timestamp}${apiSecret}`;
const signature = createHash('sha1').update(signatureBase).digest('hex');
```

Esto es correcto para un upload firmado básico de Cloudinary (parámetros en orden alfabético, concatenados con el secret al final). Si los parámetros en el FormData incluyen exactamente `folder` y `timestamp` (y nada más), la firma debe validar.

**Acción si los campos llegan bien pero la firma falla**: los valores de `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` en `backend/.env` no coinciden con la cuenta real. Verificar en [Cloudinary Console → Settings → API Keys](https://console.cloudinary.com/settings/api-keys).

## Bloque C — Nueva funcionalidad: gestión de grupos de modificadores

### Contexto del dominio

El modelo `ModifierGroup` ([schema.prisma:299](../../backend/prisma/schema.prisma#L299)) pertenece a `Organization` (campo `organizationId`), no directamente al restaurante. El restaurante accede a los grupos de su organización y los asigna a sus productos a través de `RestaurantProductModifierGroup`.

Actualmente **no existe ningún endpoint de CRUD** para modifier groups en el backend. Solo se devuelven incrustados en la respuesta del menú.

---

### C1. Backend: CRUD de grupos de modificadores

**Estrategia**: seguir la misma arquitectura limpia que `CreateRestaurantProductUseCase` y el resto de casos de uso existentes.

#### C1.a — Casos de uso

Crear en `backend/src/restaurants/application/use-cases/`:

| Archivo | Responsabilidad |
|---|---|
| `create-modifier-group.use-case.ts` | Valida nombre único en la organización, crea grupo + opciones en transacción |
| `list-modifier-groups.use-case.ts` | Lista grupos de la organización del restaurante |
| `delete-modifier-group.use-case.ts` | Solo borra si ningún `RestaurantProduct` lo usa actualmente |

> No se propone editar grupos por ahora: editar opciones de un modificador en uso puede afectar pedidos activos. Se puede añadir en una iteración posterior.

#### C1.b — Puerto nuevo

Añadir métodos en el repositorio de lectura existente **o** crear un puerto separado `ModifierGroupRepository`:

```typescript
// application/ports/modifier-group-repository.port.ts
export interface ModifierGroupRepository {
  findByOrganization(organizationId: string): Promise<ModifierGroupEntity[]>;
  findOrganizationByRestaurantId(restaurantId: string): Promise<string | null>;
  create(data: CreateModifierGroupData): Promise<ModifierGroupEntity>;
  isAssignedToAnyProduct(groupId: string): Promise<boolean>;
  delete(groupId: string): Promise<void>;
}
```

#### C1.c — Repositorio Prisma

Crear `backend/src/restaurants/infrastructure/persistence/prisma-modifier-group.repository.ts`. Necesita la query de Prisma:

```typescript

  data: {
    organizationId,

Añadir spec de integración con Testcontainers.

#### C1.d — Controlador REST

Crear `backend/src/restaurants/presentation/rest/restaurant-modifier-groups.controller.ts`:

```
POST   /api/v1/restaurants/:id/modifier-groups       → crear grupo
